# Japanese Tech Generation

## 目的

このスキルは、日本語の技術文書を新規に生成するとき、最初から論証の筋が通った本文にするために使う。
きれいな言い回しを増やすのではなく、読者が段落単位で理解を進められる文章を作る。

## 併用する規範

作業前に `../japanese-tech-writing/SKILL.md` を読む。
特に「段落と論証の構成」「論証の厳密さ」「読み手の負荷の管理」「LLM っぽい表現の禁止」「冗長の排除」に従う。

既存原稿を直す、または生成後に段落間の飛躍を点検する場合は、`../argument-gap-edit/SKILL.md` も読む。

## 適用範囲

使う場面:

- 技術書の章・節・コラムを書く
- 技術記事、解説文、設計判断の説明を書く
- 調査結果や実装方針を読者向けの本文に変換する
- メモや箇条書きから、論証を持つ日本語本文を生成する

使わない場面:

- 一、二文の短い返答
- コード、設定、コマンド出力が主役の回答
- 文章レビューだけを行う依頼
- 既存原稿の論理修正が主目的の依頼

## 生成手順

1. 読者、目的、文書種別、前提知識を一文で定める。
2. 先に結論ではなく、読者がその結論を受け入れるために必要な論理の順序を決める。
3. 各段落について、次の三つを短く決めてから本文を書く。
   - 前段落から受けるもの
   - その段落で進める論点
   - 次段落へ渡すもの
4. 本文は一段落一トピックで書く。
5. 新しい術語、理論、引用、比喩は、それが後続の判断を変える場合だけ導入する。
6. 書き終えたら、段落ごとに「受けるもの」「進める論点」「渡すもの」を再確認する。
7. 最後に、空虚な強調、同じ主張の反復、根拠のない断定、回収されない前方参照を削る。

## 生成時の優先順位

- 自然な文章より、論理の役割が明確な文章を優先する。
- 網羅より、読者が次に必要とする情報だけを優先する。
- 術語より、まず具体的な関係を優先する。
- 華やかな導入より、本文の問いがすぐ見える導入を優先する。
- まとめ直しより、本文中で一度だけ正確に言うことを優先する。

## 出力ルール

- 生成物だけを出す。依頼されていない限り、設計メモや自己点検結果は出さない。
- Markdown 文書では、一文ごとに改行し、段落の区切りは空行で示す。
- 見出しは、その節が扱う対象や問いを具体的に示す。
- 断定できない内容は、可能性・推量・仮定として正確に書く。
- 「重要なのは」「多角的に」「正面から」「深掘りする」など、内容を増やさない LLM 口調を使わない。

## 自己点検

出力前に次を確認する。

- 各段落の冒頭が、前段落の語・問い・未解決事項を受けているか。
- 例や引用が、支えたい主張と同じ問いに答えているか。
- 新しい概念を導入したあと、その概念で後続の判断を進めているか。
- 同じ対象の説明中に別の話題を割り込ませていないか。
- 最後の段落が、本文で積み上げた内容だけから出ているか。
